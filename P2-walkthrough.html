CS 4880 P2 — Code Walkthrough

Compiler Design · Lexical Analyzer + Recursive Descent Parser · Spring 2026

Overview & Architecture

P2 is a two-phase front-end of a compiler for a toy language. It is built on top of the P1 lexer and adds a recursive descent parser that builds a parse tree which is then printed to stdout. The project is structured in four clean layers with strict separation of concerns:

token.h
Layer 1
Shared data types only. No logic. The lingua franca of all layers.
lexer.h / lexer.c
Layer 2
Character scanner. Converts source text → Token stream.
tree.h / tree.c
Layer 3
Generic parse-tree node. Allocate, link, print, free.
parser.h / parser.c
Layer 4
LL(1) recursive descent. BNF grammar → Node* tree.
Why this layering? The RecursiveDescentParser.pdf doc says each nonterminal becomes a function (Layer 4), and each function should get/return tokens from the scanner without worrying about character-level details (Layer 2). Keeping token definitions separate (Layer 1) prevents circular header dependencies.

token.h — Shared Data Definitions

Layer 1 Pure header. No .c counterpart — zero logic here.

#ifndef TOKEN_H
#define TOKEN_H

typedef enum {
    IDTk,      /* user-defined identifier  */
    IntTk,     /* integer literal          */
    KW_tk,     /* language keyword         */
    OpTk,      /* multi-char operator       */
    OpDelTk,   /* single-char op/delimiter  */
    EOFTk      /* end of input              */
} tokenID;

extern const char * const tokenNames[];   /* defined in lexer.c */

#define TOKEN_INSTANCE_MAX 9   /* 8 significant chars + NUL */

typedef struct {
    tokenID  id;
    char     instance[TOKEN_INSTANCE_MAX];
    int      lineNumber;
} Token;

#endiftoken.h

Why this enum layout?

The language spec (P1/P2 BNF) only needs six categories to distinguish every possible token. The parser only ever checks tk.id plus tk.instance — it never needs to peek at raw characters again. This clean boundary is what makes the parser completely independent of the lexer's internals.

BNF Connection: The BNF grammar uses terminals like identifier, integer, go, scan, etc. These map directly onto token IDs:
  • identifier → IDTk
  • integer → IntTk
  • go, exit, int, scan, output, … → KW_tk
  • ?le, ?ge, **, // → OpTk
  • =, :, {, }, [, ], (, ), +, - → OpDelTk

The TOKEN_INSTANCE_MAX = 9 constant

The language spec says identifiers are at most 8 characters long and integers are at most 8 digits. Adding 1 for the null terminator gives 9. This single constant is used everywhere — the struct definition, strncpy calls in the lexer, and the lexer's local buf[TOKEN_INSTANCE_MAX] — so changing the limit only requires editing one line.

Just two functions. lexer_init must be called once to bind the lexer to a file. getNextToken is the scanner function referenced throughout RecursiveDescentParser.pdf — the doc calls it scanner().

lexer.c — The Lexical Analyzer

Layer 2 Stateful character-by-character scanner. Converts a stream of characters into Tokens.

#include <ctype.h>                 /* isalpha, isdigit, isspace             */

static int   lineNum = 1;lexer.c (top)

static on both variables means they have file scope only — nothing outside lexer.c can access or corrupt them. lineNum starts at 1 because the first line of a file is line 1, not 0.

nextChar and pushBack

static int nextChar(void) {
    int c = fgetc(src);
    if (c == '\n')
        lineNum++;
    return c;
}

static void pushBack(int c) {
    if (c == '\n')
        lineNum--;
    ungetc(c, src);
}lexer.c:23-36

These two wrappers around fgetc / ungetc are the only places that advance or retreat in the input. Every other function in the lexer calls these, so line tracking is handled in one place. pushBack undoes the newline count so lineNum stays accurate when the lexer "peeks" ahead one character and then decides not to consume it (e.g., seeing a lone * before confirming it's **).

Why return int, not char? fgetc returns int so EOF (typically -1) fits without conflicting with any valid unsigned char value. Storing it as char first could silently truncate the sentinel value on platforms where char is unsigned.

Comments — @...@

if (c == '@') {
    int c2;
    while (1) {
        c2 = nextChar();
        if (c2 == EOF)    lexError("unterminated comment");
        if (isspace(c2))  lexError("whitespace inside comment");
        if (c2 == '@')    break;
    }
    goto restart;  /* O(1) stack space vs. recursive call */
}lexer.c:95-109

Comments are delimited by @ on both sides and may not contain whitespace. After consuming the closing @, the lexer jumps back to the top of getNextToken via goto restart rather than calling itself recursively. The inline comment explains the reason: a sequence of back-to-back comments like @a@ @b@ @c@ would grow the C stack one frame per comment with a recursive approach; the goto keeps it O(1).

Same maximal-munch pattern as identifiers but only for digits. No floating-point, no hex, no negative sign here — negative numbers are handled at the parser level via the <N> rule's unary minus production.

tree.h — Parse Tree Node Definition

Layer 3 The data structure that the parser builds and then prints.

#define MAX_CHILDREN 4

typedef struct Node {
    char         label[32];            /* nonterminal name, e.g. "program" */
    Token        tokens[3];            /* semantic tokens stored here      */
    int          tokenCount;
    struct Node *children[MAX_CHILDREN]; /* sub-nonterminal nodes             */
    int          childCount;
} Node;tree.h

Three parallel decisions are visible in this struct, each directly traceable to the course docs:

FieldWhy that size?Doc Source
label[32]Longest nonterminal name (relational) is 10 chars; 32 is safe headroomRecursiveDescentParser.pdf p.3: "A label identifying the function that created the label (and thus nonterminal)"
tokens[3]Maximum semantic tokens in any single production — <varList> stores two (identifier + integer), <relational> stores two for = =RecursiveDescentParser.pdf p.3: "maximum number of semantics token in any production"
children[4]Maximum nonterminals on any RHS — <cond> and <loop> have 3 (<relational>, <exp>, <stat>); 4 gives safe marginRecursiveDescentParser.pdf p.3: "maximum number of nonterminal in any production"
Semantic vs. Structural tokens: The doc says "structural tokens are disposed: {}, (), delimiters" while "semantics tokens are stored in the node: IDTk, NumberTk, OperatorTk." In this code, keywords like go, scan, int are consumed and discarded (the node label already records which rule matched). Only IDTk, IntTk, and operator tokens are passed to addToken().

tree.c — Tree Operations

Layer 3 Four functions: create, attach, print, free.

newNode

Node *newNode(const char *label)
{
    Node *n = malloc(sizeof(Node));
    if (!n) { fprintf(stderr, "out of memory\n"); exit(1); }
    strncpy(n->label, label, sizeof(n->label) - 1);
    n->label[sizeof(n->label) - 1] = '\0';
    n->tokenCount = n->childCount = 0;
    for (int i = 0; i < MAX_CHILDREN; i++) n->children[i] = NULL;
    return n;
}tree.c:7-17

Allocates on the heap (malloc) so the node outlives the parser function that created it. The doc says functions return Node* pointers — stack allocation would be dangling by the time the caller adds the node as a child.

printTree — Indented Depth-First Traversal

void printTree(const Node *n, int depth)
{
    if (!n) return;
    for (int i = 0; i < depth; i++) printf("  ");   /* 2-space indent per level */
    printf("%s", n->label);
    for (int i = 0; i < n->tokenCount; i++) printToken(n->tokens[i]);
    printf("\n");
    for (int i = 0; i < n->childCount; i++) printTree(n->children[i], depth + 1);
}tree.c:46-54

Pre-order traversal (print node, then recurse into children). Each level gets 2 extra spaces, so the visual tree depth matches structural depth. Tokens are printed inline with the label: ID:xVar:3, #:42:5, Op:+:5.

freeTree — Post-Order Cleanup

void freeTree(Node *n) {
    if (!n) return;
    for (int i = 0; i < n->childCount; i++) freeTree(n->children[i]);
    free(n);
}tree.c:56-61

Post-order: free all children before freeing the parent. This prevents use-after-free — if the parent were freed first its child pointers would be dangling.

parser.h — Parser Interface

Layer 4 Minimal: one public function.

void parser(void);parser.h

main calls parser(), the entire recursive descent parse + tree build + tree print happens inside, and control returns. The driver doesn't know or care about Node*, tokens, or any grammar detail.

parser.c — Recursive Descent Parser

Layer 4 The heart of P2. One C function per BNF nonterminal.

Key principle from RecursiveDescentParser.pdf: "Every nonterminal is implemented as a function … Each function is called with unconsumed token and returns with unconsumed token." This means the global tk always holds the next unprocessed token. When a function needs to verify or consume it, it checks tk then calls consume().

Helper Functions

static Token tk;     /* the one lookahead token — global to parser.c */

static void error(const char *expected) {
    fprintf(stderr, "SYNTAX ERROR: expected %s, got '%s' (line %d)\n",
            expected, tk.instance, tk.lineNumber);
    exit(1);
}

static void consume(void) { tk = getNextToken(); }

static int isKW   (const char *kw) { return tk.id == KW_tk    && strcmp(tk.instance, kw) == 0; }
static int isOpDel(const char *op) { return tk.id == OpDelTk  && strcmp(tk.instance, op) == 0; }
static int isOp   (const char *op) { return tk.id == OpTk     && strcmp(tk.instance, op) == 0; }parser.c:11-55

consume() is exactly tk = scanner() from the doc's pseudocode. The three is*() predicates check both the token category (.id) and the exact string (.instance) because the parser must distinguish = (OpDelTk) from ?eq (OpTk), both of which are "operator-like."

isStatFirst — FIRST set for <stat>

static int isStatFirst(void) {
    return isKW("scan")   || isKW("output") || isKW("cond") ||
           isKW("loop")   || isKW("set")    || isOpDel("{");
}parser.c:57-61
LL(1) FIRST set: The doc says "predict one production based on the current token … select the production whose First set contains the token." FIRST(<stat>) = { scan, output, cond, loop, set, { } — one keyword or { unambiguously identifies which production to take. isStatFirst is used by mStat() to decide whether the empty production applies.

program / vars / varList / block

program_rule — BNF: <program> → go <vars> <block> exit

static Node *program_rule(void)
{
    Node *n = newNode("program");
    if (!isKW("go")) error("'go'");
    consume();              /* consume 'go' — structural, not stored */
    addChild(n, vars());
    addChild(n, block());
    if (!isKW("exit")) error("'exit'");
    consume();              /* consume 'exit' — structural */
    return n;
}parser.c:66-76

Single production — no branching needed. Just process elements left-to-right: terminal go is checked and consumed; nonterminals vars and block are called and their returned nodes become children; terminal exit is checked and consumed.

vars — BNF: <vars> → empty | int identifier = integer <varList> :

static Node *vars(void)
{
    if (!isKW("int")) return NULL;   /* empty production */
    Node *n = newNode("vars");
    consume();                         /* int — structural */
    if (tk.id != IDTk) error("identifier after 'int'");
    addToken(n, tk);  consume();        /* identifier — semantic, stored */
    if (!isOpDel("=")) error("'='");
    consume();                         /* = — structural */
    if (tk.id != IntTk) error("integer");
    addToken(n, tk);  consume();        /* integer — semantic, stored */
    addChild(n, varList());
    if (!isOpDel(":")) error("':'");
    consume();                         /* : — structural */
    return n;
}parser.c:79-96

Two-production rule. The discriminator is whether tk is the keyword int: if not, take the empty production and return NULL. The doc says a function returning on the empty production can return null. addChild in tree.c guards against null children, so parent nodes do not need to check.

varList — BNF: <varList> → identifier = integer <varList> | empty

static Node *varList(void)
{
    Node *n = newNode("varList");
    if (tk.id != IDTk) return n;    /* empty production — return empty node */
    addToken(n, tk);  consume();
    if (!isOpDel("=")) error("'='");
    consume();
    if (tk.id != IntTk) error("integer");
    addToken(n, tk);  consume();
    addChild(n, varList());           /* right-recursive — handles list of any length */
    return n;
}parser.c:99-112

Unlike vars(), this always allocates the node (even on empty) before checking whether to recurse. Right-recursion is safe here because the grammar is LL(1) and right recursion does not require left-factorization.

stats / mStat / stat

These three rules implement the "one or more statements" idiom without using iteration in C.

stats — BNF: <stats> → <stat> <mStat>

static Node *stats(void) {
    Node *n = newNode("stats");
    addChild(n, stat());    /* mandatory: at least one statement */
    addChild(n, mStat());   /* optional: zero or more additional  */
    return n;
}parser.c:127-134

mStat — BNF: <mStat> → empty | <stat> <mStat>

static Node *mStat(void) {
    Node *n = newNode("mStat");
    if (!isStatFirst()) return n;   /* empty production */
    addChild(n, stat());
    addChild(n, mStat());           /* right-recursion for next statement */
    return n;
}parser.c:136-144
Why right-recursion instead of a loop? The doc says to implement the grammar directly as written using recursive functions, exploiting the C call stack as the implicit parse stack. A C while loop could work here but would not match the BNF, and the doc explicitly says each nonterminal gets its own function. The right-recursive structure also naturally builds a right-leaning subtree that represents the order of statements.

stat — dispatches by FIRST token

static Node *stat(void) {
    Node *n = newNode("stat");
    if      (isKW("scan"))   addChild(n, read());
    else if (isKW("output")) addChild(n, print());
    else if (isOpDel("{"))   addChild(n, block());
    else if (isKW("cond"))   addChild(n, cond());
    else if (isKW("loop"))   addChild(n, loop());
    else if (isKW("set"))    addChild(n, assign());
    else                     error("statement keyword or '{'");
    return n;
}parser.c:146-158

This is the canonical LL(1) dispatch table in code form. Each branch's FIRST set is disjoint (enforced by the language design): scan, output, {, cond, loop, set are all unique tokens, so exactly one branch is chosen with one token of lookahead — hence LL(1).

    consume();                      /* cond */

    consume();                      /* [ */

    addChild(n, relational());
    addChild(n, expr());

relational — restricting to the BNF operators, and the two-token = = case

/* <relational> accepts ONLY the four ?-ops named in the BNF.
   ?gt and ?ne are valid P1 tokens but are NOT relational operators. */
static int isRelOp(void) {
    return tk.id == OpTk &&
           (strcmp(tk.instance, "?le") == 0 || strcmp(tk.instance, "?ge") == 0 ||
            strcmp(tk.instance, "?lt") == 0 || strcmp(tk.instance, "?eq") == 0);
}

static Node *relational(void) {
    Node *n = newNode("relational");
    if (isRelOp()) {                     /* ?le ?ge ?lt ?eq */
        addToken(n, tk); consume();
    } else if (isOpDel("=")) {         /* first '=' of = = */
        addToken(n, tk); consume();
        if (!isOpDel("=")) error("second '=' in '= ='");
        addToken(n, tk); consume();  /* second '=' */
    } else if (isOpDel(";")) {
        addToken(n, tk); consume();  /* the semicolon relational */
    } else error("relational operator");
    return n;
}parser.c:237-263
Why isRelOp() instead of tk.id == OpTk: The lexer recognizes six ?-operators (?le ?ge ?lt ?gt ?ne ?eq), but the BNF's <relational> production lists only four. A bare tk.id == OpTk test would also accept ?gt, ?ne, and even the expression operators ** / // in the comparison position — so cond [ x ** 0 ] would parse instead of erroring. The explicit instance check matches the grammar exactly.

= = is two separate OpDelTk("=") tokens. The parser reads the first =, stores it, then checks the second. Both are stored in the node because both are semantic — they collectively represent the equality operator and a later phase needs to know what the operator was. The ; alternative is likewise a single stored OpDelTk.

Expression Rules — The Delay Technique

The three expression nonterminals encode arithmetic precedence directly in the grammar structure. No explicit precedence table is needed.

<exp>→<M> ** <exp> | <M> // <exp> | <M>
<M>→<N> + <M> | <N>
<N>→<R> - <N> | - <N> | <R>
<R>→( <exp> ) | identifier | integer

Precedence is lowest-to-highest as you go deeper: exp handles **/// (lowest), M handles +, N handles - (unary and binary), R handles atoms and parenthesized sub-expressions (highest). A deeply nested operand is evaluated first because it is deepest in the call stack. 

/* exp — ** and // have lowest precedence among operators */
static Node *expr(void) {
    Node *n = newNode("exp");
    addChild(n, M());
    if (isOp("**") || isOp("//")) {
        addToken(n, tk);  /* store the operator */
        consume();
        addChild(n, expr()); /* right-recursive */
    }
    return n;
}

/* N — handles both binary subtraction and unary negation */
static Node *N(void) {
    Node *n = newNode("N");
    if (isOpDel("-")) {          /* unary: - <N> */
        addToken(n, tk); consume();
        addChild(n, N());
    } else {
        addChild(n, R());       /* read atom first */
        if (isOpDel("-")) {      /* binary: <R> - <N> */
            addToken(n, tk); consume();
            addChild(n, N());
        }
    }
    return n;
}

/* R — base case: parenthesized expression, identifier, or integer */
static Node *R(void) {
    Node *n = newNode("R");
    if (isOpDel("(")) {
        consume();
        addChild(n, expr());       /* recurse back into full expression */
        if (!isOpDel(")")) error("')'");
        consume();
    } else if (tk.id == IDTk) {
        addToken(n, tk); consume();
    } else if (tk.id == IntTk) {
        addToken(n, tk); consume();
    } else error("identifier, integer, or '('");
    return n;
}parser.c:258-322
The "delay technique": Each expression function calls into the next-higher-precedence function first (e.g., expr calls M() before checking for **). This means the left operand is always fully parsed before the operator is consumed. This is the standard LL grammar technique for handling operators without left recursion.

parser() — the auxiliary entry point

void parser(void)
{
    tk = getNextToken();      /* prime the lookahead */
    Node *tree = program_rule();
    if (tk.id != EOFTk)
        error("end of input");   /* trailing garbage */
    printTree(tree, 0);
    freeTree(tree);
}parser.c:325-333

This matches the doc's "auxiliary function" pattern exactly: get first token, call the start nonterminal (program_rule), then verify EOFTk. After verifying the full tree is correct it prints then frees — the print must come first because freeing first would invalidate all the pointers.

P2.c — The Driver

{
    FILE *in = NULL;

    lexer_init(in);

    if (lexOnly) runLexer();
    else         parser();

    if (in != stdin) fclose(in);
    return 0;
}P2.c

The driver does three things only: file handling, mode selection, and delegation. The -l flag is a backward-compatibility shim to run the P1 lexer dump (useful for debugging lexer issues without invoking the parser). The .ext suffix is appended automatically so you can type ./P2 test_good1 instead of ./P2 test_good1.ext.

Why LL(1)?

LL(1) means the parser reads input Left-to-right, constructs a Leftmost derivation, using 1 token of lookahead. The grammar meets LL(1) requirements because:

  1. No left recursion — all recursive calls to the same nonterminal are on the right side (e.g., <varList> → id = int <varList>).
  2. Disjoint FIRST sets — for every nonterminal with multiple productions, the leading tokens of each alternative are unique. For example, FIRST(scan identifier :) = {scan} is disjoint from FIRST(output <exp> :) = {output}.
  3. FIRST / FOLLOW disjoint for ε-productions — nonterminals that can derive empty (<vars>, <varList>, <mStat>) have their FIRST sets disjoint from their FOLLOW sets, so the parser always knows whether to take the empty branch.

The Lookahead Pattern

The global static Token tk is the single lookahead. The rule throughout the entire parser is: tk always holds the next unconsumed token.

parser() calls getNextToken()
→
tk = first token
→
call program_rule()
→
check tk, consume(), recurse…

When a function returns, it has left tk pointing at the token after everything it consumed. The caller can immediately inspect tk to decide what comes next. No buffering, no token stack — the C call stack itself is the parse stack (this is what the doc means when it says function calls are activation record pushes).

Tree Generation Strategy

The doc separates tokens into two categories and the code follows this exactly:

Token TypeTreatmentExample
StructuralConsumed and discarded (not stored in node)go exit int { } [ ] ( ) : =
SemanticStored via addToken(n, tk)IDTk IntTk OpTk OpDelTk(+,-)

This means a later compiler phase (semantic analysis, code generation) has everything it needs in the tree without needing to re-read the source. The identifier name, the integer value, and the operator are all attached to the relevant node.
